Fabricate Studio Panel Guide

Learn how to use the Fabricate Studio panel to manage test data, scenarios, and field generation. This guide is for everyone — PO/QA testing user flows, and developers working offline during development.

No API required PO friendly Live mapping

1. Seed scenarios

Use preset buttons first. “Happy path demo” fills users/profile. “Empty state” resets data. “Faker mapping demo” injects dynamic generated fields.

2. Validate screens

Go back to the app and run user journeys. Since storage updates live, UI screens react instantly without redeploying anything.

3. Tune generated data

Use Simple mode for form-based mapping. Switch to Advanced JSON mode only when you need full manual control.

Simple mode vs advanced mode

Simple mode (recommended for PO/QA)Add fields with the form builder: choose route, field path, type, and options. No raw JSON needed.

Advanced JSON modeUse only for power users. You can still edit the complete mapping object manually.

Storage tab

What it showsThe Storage tab displays all test data currently stored in localStorage. Each key is prefixed with fabricate: and includes the route path.

Collection formatArrays stored with key pattern: fabricate:collection:/pathfabricate:collection:/users = [ { id: "u-1", name: "Ada", role: "admin" }, { id: "u-2", name: "Grace", role: "user" } ]

Resource formatSingle objects stored with key pattern: fabricate:resource:/pathfabricate:resource:/profile = { id: "me", firstName: "John", role: "admin" }

Use during testingrefresh the page and click "Storage" to verify your scenario data persisted. Edit values directly if needed to test edge cases.

What is Faker mapping?

Core conceptFaker mapping automatically generates realistic test data based on rules you define. When you POST data to an endpoint, the stub engine adds extra fields automatically according to your mapping configuration.

Real world exampleYou only provide a minimal request payload:// Your POST request body { "name": "Alice" }Faker mapping intercepts and auto-generates:// Response with auto-generated fields { "name": "Alice", "age": 34, "displayName": "Alice Johnson", "role": "user", "createdAt": "2025-10-01" }

Use caseTest the UI with varied, realistic data without manually crafting each test case. Especially useful for testing forms, lists, and edge cases (boundary ages, special characters, timestamps).

How it worksIn the Mappings tab, define rules like:{ "/users": { "displayName": { "type": "person.fullName" }, "age": { "type": "number.int", "min": 18, "max": 65 } } }When a POST happens, these rules auto-generate values for those fields.

Faker mapping examples

Age between 18 and 30 (string){ "/users": { "age": { "type": "number.int", "min": 18, "max": 30, "asString": true } } }

Random names and category list{ "/users": { "displayName": { "type": "person.fullName" }, "category": { "type": "pick", "values": ["car", "peter", "house"] } } }

Recent timestamp{ "/users": { "createdAt": { "type": "date.recent", "days": 7 } } }

Nested objects in Simple mode

How to do itIn field path, use dot notation. Example: profile.firstName, profile.lastName, profile.meta.createdAt.

Form builder inputsRoute: /users Field path: profile.firstName Type: person.firstName Field path: profile.lastName Type: person.lastName Field path: profile.meta.createdAt Type: date.recent Options: days = 7

Generated mapping{ "/users": { "profile": { "firstName": { "type": "person.firstName" }, "lastName": { "type": "person.lastName" }, "meta": { "createdAt": { "type": "date.recent", "days": 7 } } } } }

Recommended QA workflow

Step AOpen Stub Panel and click “Happy path demo”.

Step BExecute main functional flow in the app.

Step CSwitch to “Empty state” and retest empty UX.

Step DEdit faker mapping, trigger POST flow, verify generated values in UI.
